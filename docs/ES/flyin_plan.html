Día 2 · Algoritmo de Rutas
BFS / Dijkstra adaptado al proyecto MEDIUM
Pathfinding single-drone · Costes por tipo de zona · Sin networkx
7–9h
horas
▶
Teoría a aprender
  • BFS (Breadth-First Search): cómo encuentra el camino más corto en grafos no ponderados
  • Dijkstra: camino mínimo en grafos con pesos. Implementación con heapq
  • Cola de prioridad en Python: import heapq — cómo funciona
  • Costes del proyecto: normal=1, restricted=2, priority=1 (preferida), blocked=∞
  • Diferencia entre "coste de arista" y "coste de nodo destino"
  • Por qué NO puedes usar networkx (obligatorio implementar tú)
Código a escribir
  • Método get_neighbors(zone) en Graph
  • Función get_movement_cost(zone_type) → int
  • Clase PathFinder con método find_path(start, end) → List[Zone]
  • Implementar Dijkstra desde cero con heapq (sin networkx)
  • Manejar zonas blocked: simplemente no añadirlas como vecinas válidas
  • Manejar zonas priority: coste 1 pero priorizarlas en desempate
Conceptos críticos del proyecto
  • Las zonas blocked son INACCESIBLES — nunca aparecen en un camino
  • Una zona restricted cuesta 2 turnos — el drone ocupa la conexión 1 turno
  • Una zona priority cuesta 1 turno pero debe ser PREFERIDA ante igualdad
  • El coste está en el DESTINO, no en la conexión (salvo max_link_capacity)
  • El pathfinder de hoy solo planifica 1 drone — el día 3 maneja N drones

Día 3 · El Corazón del Proyecto
Motor de Simulación por Turnos HARD
Multi-drone · Capacidades · Detección de conflictos · Formato de salida
8–10h
horas
▶
Conceptos clave a entender
  • Simulación discreta por turnos: qué pasa EXACTAMENTE cada turno
  • Ocupación de zonas: los drones que SALEN liberan espacio ese mismo turno
  • max_drones por zona: cuántos pueden estar simultáneamente
  • max_link_capacity: cuántos pueden cruzar la misma conexión a la vez
  • Drones en tránsito hacia restricted: ocupan la conexión 1 turno extra
  • Deadlock: dos drones bloqueándose mutuamente — cómo detectarlo y evitarlo
  • Start y End son especiales: capacidad ilimitada
Código a escribir
  • Clase Drone: id, posición actual, camino asignado, estado (moving/waiting/transit)
  • Clase SimulationState: turno actual, posición de cada drone
  • Clase Simulator: ejecuta la simulación turno a turno
  • Método compute_turn(): calcular movimientos válidos de todos los drones
  • Método check_capacity(zone, turn): verificar si hay espacio
  • Detección de drones en tránsito (restricted zones)
  • Formato de salida: D1-zona D2-zona\n por cada turno
Las reglas más complicadas (¡no las olvides!)
  • Drones que SALEN de una zona la liberan en ESE MISMO turno (no el siguiente)
  • Un drone en tránsito a restricted: DEBE llegar el siguiente turno, no puede esperar en la conexión
  • Drones que ya llegaron a end_hub desaparecen — ya no se imprimen
  • Si un drone no se mueve en un turno, NO aparece en la línea de salida
  • La simulación termina cuando TODOS los drones han llegado al end
Entregable al final del día

  • ✅ Respeta max_drones por zona y max_link_capacity por conexión
  • ✅ Los 3 mapas easy funcionan dentro del target (≤6, ≤6, ≤8 turnos)
  • ✅ Drones que esperan no aparecen en el output
  • ✅ Zonas restricted con tránsito de 2 turnos funcionando

Asignación de Rutas Múltiples + Scheduling HARD

horas

Teoría a aprender
  • Flujo máximo en grafos (concepto teórico, no implementar completo)
  • Edge-disjoint paths: caminos que no comparten aristas — útil para N drones simultáneos
  • Staggering: escalonar la salida de drones para evitar cuellos de botella
  • Greedy scheduling: asignar drones al camino más libre disponible

  • Dead-end detection: detectar si un camino es trampa antes de enviarlo

  • Refactorizar Simulator para usar asignación pre-calculada de rutas
  • Detección y exclusión de dead-ends antes de la simulación
  • Cache de rutas calculadas para no recalcular en cada turno
Los medium maps que debes superar hoy

  • circular_loop (6 drones, ≤20 turnos): usa las zonas priority correctamente
  • priority_puzzle (4 drones, ≤12 turnos): priority debe ser PREFERIDA
  • Si alguno falla el target: debuggear el scheduling, no el pathfinder

// Funciones, librerías y prácticas — permitidas vs prohibidas
✓ PERMITIDO
Librerías estándar de Python
heapq collections typing dataclasses sys os re math copy enum abc itertools functools pathlib argparse logging
Librerías externas (sí permitidas)
matplotlib — visualización gráfica pygame — GUI opcional pytest — tests (no se entrega) flake8 — linting obligatorio mypy — type checking obligatorio
Prácticas correctas
  • ✓ Implementar Dijkstra/BFS desde cero con heapq
  • ✓ Usar dict / list para representar el grafo
  • ✓ Type hints en todas las funciones y variables
  • ✓ Docstrings en todas las clases y métodos (PEP 257)
  • ✓ try/except para manejar errores de I/O y parseo
  • ✓ Context managers (with open(...)) para abrir ficheros
  • ✓ Clases OOP con responsabilidad única por clase
  • ✓ Usar @dataclass para clases de datos simples
  • ✓ Usar IA para entender conceptos, comprobar lógica — no para copiar sin entender
  • ✓ Crear tus propios mapas de test para edge cases
✗ PROHIBIDO
Librerías de grafos — BAN total
networkx graphlib igraph graph-tool scipy.sparse.csgraph pyvis rustworkx cualquier otra que haga pathfinding
Prácticas prohibidas en el código
  • ✗ Variables sin type hint en funciones (mypy lo pillará)
  • ✗ Funciones sin docstring (se penaliza en peer review)
  • ✗ Código que no pase flake8 — líneas largas, imports no usados
  • ✗ Variables globales mutables fuera de clases
  • ✗ Código procedural sin OOP — todo debe estar en clases
  • ✗ Abrir ficheros sin context manager (open() suelto sin with)
  • ✗ Excepciones sin manejar que crashen el programa en review
  • ✗ Nombres de zona con guiones en los mapas propios que crees
  • ✗ Dos conexiones a→b y b→a en el mismo fichero (duplicado)
Lo que NO puedes hacer con IA (según el enunciado)
  • ✗ Copiar una función completa de IA sin entender cada línea
  • ✗ Pegar código de IA que no puedas explicar en peer review
  • ✗ Usar IA como sustituto de entender el algoritmo — si fallas la explicación, fallas el proyecto
⚠ ZONA GRIS — Depende de cómo lo uses
numpy / scipy
Permitido si no usas funciones de pathfinding. Si solo lo usas para arrays o cálculos matemáticos auxiliares, está bien. Pero si usas scipy.sparse.csgraph.shortest_path → prohibido.
collections.deque
Perfectamente válido para implementar BFS tú mismo. La deque es una estructura de datos estándar, no una librería de grafos. Úsala sin miedo en tu PathFinder.
Código generado por IA
Permitido si lo entiendes, lo revisas con un peer, y puedes explicarlo. El enunciado lo dice explícitamente: úsala para reducir tareas repetitivas, no como sustituto de pensar.
